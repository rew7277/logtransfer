Overview

Getting Started
Quickstart

Get logs into ObserveX in under 2 minutes. No email verification required.

01

Create a workspace

Visit /create-account, enter your organization name, email, and password. Your workspace URL is generated instantly at /workspace/your-slug.

02

Generate an API key

Go to Ingest API → Your API keys → New key. Give it a name like production. The full key is shown once — copy it immediately.

03

Send your first log

bash
curl -X POST https://yourapp.railway.app/ingest/json \ -H "Authorization: Bearer oxk_your_key_here" \ -H "Content-Type: application/json" \ -d '{"timestamp":"2024-01-15T10:30:00Z","level":"error","source":"my-service","message":"Connection refused"}'
04

Open the Log Explorer

Go to your workspace and click Log explorer in the sidebar. Your log should appear immediately. Use the search and level filters to find it.

05

Create your first alert

Go to Alert rules → Create Alert Rule. Set type to Error Rate, threshold to 5, add your email, and save. You'll be notified whenever errors spike.

API Reference
Authentication

All API requests must include a Bearer token in the Authorization header. Get your token from Ingest API → API keys in your workspace.

⚠
API keys are shown only once at creation. Store them securely — they cannot be retrieved after leaving the page. Revoke and regenerate if lost.
Header format
Authorization: Bearer oxk_your_api_key_here
API key properties
PropertyDescription
prefixFirst 8 characters visible for identification (e.g. oxk_a8f3)
nameHuman-readable label you set at creation
statusactive or revoked
last_used_atTimestamp of most recent successful auth

Ingest API

Push log events into ObserveX via HTTP from any service, script, or integration.

Endpoints
POST
/ingest/json
Ingest a single JSON log event or an array of events
POST
/ingest/batch
Ingest up to 1,000 events in one request via {"events":[...]}
POST
/ingest/file
Upload a log file via multipart/form-data
Single event — request body fields
FieldTypeRequiredDescription
timestampstringYesISO 8601 datetime (e.g. 2024-01-15T10:30:00Z)
levelstringYesOne of: error warn info success debug
sourcestringYesSource service or component name (e.g. payment-service)
messagestringYesHuman-readable log message
event_idstringNoYour own correlation/trace ID
*anyNoAny additional fields are stored in the payload JSON column
Example — single event
curl
curl -X POST https://yourapp.railway.app/ingest/json \ -H "Authorization: Bearer oxk_your_key_here" \ -H "Content-Type: application/json" \ -d '{ "timestamp": "2024-01-15T10:30:00Z", "level": "error", "source": "payment-service", "message": "Stripe timeout after 30s", "requestId": "req_abc123", "userId": "usr_456", "latency_ms": 30042 }'
Example — batch ingest
curl
curl -X POST https://yourapp.railway.app/ingest/batch \ -H "Authorization: Bearer oxk_your_key_here" \ -H "Content-Type: application/json" \ -d '{ "events": [ {"timestamp":"2024-01-15T10:30:00Z","level":"info","source":"api","message":"Order placed","orderId":"ORD-001"}, {"timestamp":"2024-01-15T10:30:01Z","level":"error","source":"api","message":"Payment failed","code":402} ] }'
Example — file upload
curl
curl -X POST https://yourapp.railway.app/ingest/file \ -H "Authorization: Bearer oxk_your_key_here" \ -F "file=@/var/log/myapp.log"
Response
{ "status": "ok", "records": 1, "source": "api-ingest", "upload_id": 42 }

Integrations
S3 Bucket Integration

Connect ObserveX to any AWS S3 (or S3-compatible) bucket to automatically pull and index log files on a schedule.

Required IAM policy

Create an IAM user with programmatic access and attach this policy (replace your-log-bucket with your actual bucket name):

IAM Policy JSON
{ "Version": "2012-10-17", "Statement": [ { "Effect": "Allow", "Action": ["s3:GetObject", "s3:ListBucket"], "Resource": [ "arn:aws:s3:::your-log-bucket", "arn:aws:s3:::your-log-bucket/*" ] } ] }
Supported file formats
FormatExtensionsNotes
JSON Lines.jsonl, .ndjsonOne JSON object per line — most efficient
CloudWatch Export.jsonAWS CloudWatch log export format
Plain text.log, .txtOne log entry per line
CSV.csvMust have header row; timestamp, level, message columns auto-detected
Gzip compressed.gzAny of the above formats compressed
Setup steps
01

Add the integration

Go to Integrations → Add integration, select S3 Bucket, enter bucket name, region, optional prefix, and IAM credentials.

02

Create a scheduled job

Go to Ingestion Jobs → Create job, select your new S3 integration, and set a cron schedule.

03

Monitor runs

Job run history is visible in Ingestion Jobs → Recent run history. Each run shows record count, status, and any error messages.

Common cron schedules
*/15 * * * * → every 15 minutes 0 * * * * → every hour 0 */6 * * * → every 6 hours 0 0 * * * → daily at midnight UTC 0 8 * * 1-5 → weekdays at 8 AM UTC

ErrorCauseFix

NoSuchBucketWrong bucket name or region mismatchVerify bucket name and region in AWS console
InvalidCredentialsWrong Access Key ID or Secret Access KeyGenerate a new IAM access key pair

Integrations
Webhooks

ObserveX can deliver alert notifications to any HTTP endpoint — Slack, Microsoft Teams, Discord, PagerDuty, or your own service.

Webhook payload

When an alert fires, ObserveX sends a POST request with this JSON body:

{ "alert_name": "Payment timeout spike", "severity": "sev2", "alert_type": "keyword", "condition": "keyword:\"Connect timeout\", threshold:5, cooldown:15m", "triggered_at": "2024-01-15T10:35:00Z", "match_count": 12, "org": "your-org-slug", "link": "https://yourapp.railway.app/workspace/your-org/explorer" }
Slack setup
01

Create a Slack app

Go to api.slack.com/apps → Create new app → From scratch → add an Incoming Webhook.

02

Get the webhook URL

Activate Incoming Webhooks, click Add New Webhook to Workspace, select a channel, and copy the URL (starts with https://hooks.slack.com/services/…).

03

Paste into alert rule

When creating or editing an alert rule, paste the Slack webhook URL into the Webhook URL field.

Microsoft Teams setup

In Teams: go to a channel → Connectors → Incoming Webhook → Configure. Copy the webhook URL and paste it into the alert rule's Webhook URL field.

Features
Alert Rules

Alert rules continuously monitor incoming log events and fire notifications when thresholds are crossed. Configure alerts in Alert rules → Create Alert Rule.

Alert types
TypeTrigger conditionUse case
Error RateError count exceeds threshold in rolling 5-minute windowDetect service outages
Keyword MatchA specific phrase appears ≥ threshold times in log messagesDetect specific exceptions, timeouts
Suspicious IPA single IP exceeds threshold req/min with elevated error rateDetect brute-force or scraping
Source Error SpikeA specific source emits more than threshold errors in 5 minDetect single-service degradation
Severity levels
LevelLabelRecommended notification
sev3MediumSlack channel message
sev2HighEmail + Slack
sev1CriticalPagerDuty / phone / all channels
Cooldown period

The Cooldown field (in minutes) prevents alert fatigue by suppressing repeated notifications for the same rule within the cooldown window. Set to 0 to fire on every occurrence. Recommended: 15–60 minutes for production SEV2/SEV1 rules.

Preview match count

Before saving a rule, click 🔍 Preview match count to see how many existing logs would currently trigger it. This lets you tune your threshold before the rule goes live.

Create alert via API
POST /api/alerts
{ "name": "Payment timeout spike", "severity": "sev2", "alert_type": "keyword", "keyword": "Connect timeout", "source_filter": "payment-service", "threshold": 5, "cooldown_minutes": 15, "notify_email": "oncall@company.com", "webhook_url": "https://hooks.slack.com/services/...", "status": "active" }
Features
✦ AI Explain

ObserveX integrates with Claude AI (Anthropic) to explain error and warning logs in plain English. Click any ERROR or WARN log in the explorer, then hit ✦ AI Explain in the detail panel.

What it explains
  • Root cause of the error in plain language
  • Likely contributing factors based on the payload
  • Suggested next steps to investigate or fix
  • Related patterns to check in other logs
Setup
ℹ
AI Explain requires an ANTHROPIC_API_KEY environment variable set on your Railway deployment. Add it in Railway → Your service → Variables.
Via API
POST /api/logs/:id/explain
curl -X POST https://yourapp.railway.app/api/logs/1042/explain \ -H "Authorization: Bearer oxk_your_key_here" # Response: { "explanation": "This error indicates a TCP connection timeout to Stripe's API...", "log_id": 1042 }

Features
Bulk Operations

Bulk mode lets you select and delete multiple log entries at once.

01

Enable bulk mode

Click ☐ Bulk in the top-right of the log explorer toolbar. Checkboxes appear on every row.

02

Select logs

Click individual checkboxes or use Select all at the top to select all logs on the current page.

03

Delete

Click 🗑 Delete selected. A confirmation dialog shows the count — confirm to permanently remove the logs.

⛔
Bulk delete is permanent and irreversible. Logs deleted this way are gone immediately. Use with caution in production workspaces.
