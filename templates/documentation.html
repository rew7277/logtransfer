Overview #

ObserveX is a centralized log management and observability platform. It collects logs from multiple sources, indexes them for real-time search, fires configurable alerts, and surfaces AI-powered error analysis — all from a single workspace URL.

📥
Multi-format Ingestion
JSON, CSV, plain text, NDJSON, MuleSoft — auto-detected from content.
🪣
S3 Integration
Poll any S3-compatible bucket on a configurable cron schedule.
🔔
Smart Alerts
Email + webhook (Slack, Teams, Discord) notifications with cooldown.
✦
AI Explain
Claude AI analyzes error logs and explains root causes in plain English.
🔍
Live Explorer
Search, filter, sort and live-tail logs with saved search support.
👥
Team Access
Role-based access with invite links and a full audit trail.
☐
Bulk Operations
Select and delete multiple logs at once in bulk mode.
📋
Audit Trail
Every admin action logged with actor, timestamp, and diff.
Getting Started
Quickstart #

Get logs into ObserveX in under 2 minutes.

01

Create a workspace

Visit /create-account, enter your org name, email, and password. Your workspace URL is generated instantly.

02

Generate an API key

Go to Ingest API → API keys → New key. The full key is shown once only — copy it to a secret manager immediately.

03

Send your first log

curl -X POST https://yourapp.railway.app/ingest/json \
  -H "Authorization: Bearer oxk_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{"timestamp":"2024-01-15T10:30:00Z","level":"error","source":"my-service","message":"Connection refused"}'
import requests

requests.post(
    "https://yourapp.railway.app/ingest/json",
    headers={"Authorization": "Bearer oxk_your_key_here"},
    json={
        "timestamp": "2024-01-15T10:30:00Z",
        "level": "error",
        "source": "my-service",
        "message": "Connection refused"
    }
)
await fetch("https://yourapp.railway.app/ingest/json", {
  method: "POST",
  headers: {
    "Authorization": "Bearer oxk_your_key_here",
    "Content-Type": "application/json"
  },
  body: JSON.stringify({
    timestamp: new Date().toISOString(),
    level: "error",
    source: "my-service",
    message: "Connection refused"
  })
});
04

Open the Log Explorer

Go to your workspace → Log explorer. Your log appears immediately. Use level chips and the search box to filter.

05

Create your first alert

Go to Alert rules → Create Alert Rule. Set type to Error Rate, threshold 5, add your email. You'll be notified whenever errors spike above 5 in a 5-minute window.

Core Concepts #
Workspace#

A workspace is an isolated tenant within ObserveX. All logs, API keys, alert rules, and team members belong to exactly one workspace. The workspace slug is globally unique and appears in every URL.

Uploads vs Events#

Each call to /ingest/* creates an upload record that groups the ingested log events. Events are queryable individually; uploads provide provenance (source IP, ingest method, timestamp, record count).

Source tag#

The source field is a free-form string identifying the emitting service or component (e.g. payment-service, 10.1.7.84). It is indexed and filterable. Use consistent source names across deployments for effective alert rules.

Payload column#

Any fields beyond the four required ones (timestamp, level, source, message) are stored verbatim in the payload JSONB column. This is surfaced in the log detail panel and sent to AI Explain for richer root-cause analysis.

API Reference
Authentication #

All API requests must include a Bearer token in the Authorization header.

⚠
API keys are shown only once at creation. Store them in a secrets manager (Doppler, Vault, Railway Variables). They cannot be retrieved after leaving the page. Revoke and regenerate if lost.
Header format
HTTP Header
Authorization: Bearer oxk_your_api_key_here
API key properties
PropertyTypeDescription
idintegerUnique identifier of the key record
prefixstringFirst 8 chars visible for identification e.g. oxk_a8f3
namestringHuman-readable label set at creation
statusstringactive | revoked
last_used_atdatetimeTimestamp of most recent successful auth (UTC)
created_atdatetimeCreation timestamp
ℹ
API keys do not have scopes currently — each key grants full API access for the workspace it belongs to. Scope-based keys are on the roadmap.
API Reference
Ingest API #

Push log events into ObserveX via HTTP from any service, script, or integration.

Endpoints
POST /ingest/json Single event or array of events (JSON body)

Accepts a single event object or a JSON array of events. Content-Type must be application/json.

curl -X POST https://yourapp.railway.app/ingest/json \
  -H "Authorization: Bearer oxk_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{
    "timestamp": "2024-01-15T10:30:00Z",
    "level": "error",
    "source": "payment-service",
    "message": "Stripe timeout after 30s",
    "requestId": "req_abc123",
    "latency_ms": 30042
  }'
import requests, datetime

resp = requests.post(
    "https://yourapp.railway.app/ingest/json",
    headers={"Authorization": "Bearer oxk_your_key_here"},
    json={
        "timestamp": datetime.datetime.utcnow().isoformat() + "Z",
        "level": "error",
        "source": "payment-service",
        "message": "Stripe timeout after 30s",
        "requestId": "req_abc123",
        "latency_ms": 30042
    }
)
print(resp.json())  # {"status":"ok","records":1}
POST /ingest/batch Up to 1,000 events in a single request

Wrap events in {"events": [...]}. Returns count of successfully ingested records.

curl
curl -X POST https://yourapp.railway.app/ingest/batch \
  -H "Authorization: Bearer oxk_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{
    "events": [
      {"timestamp":"2024-01-15T10:30:00Z","level":"info","source":"api","message":"Order placed","orderId":"ORD-001"},
      {"timestamp":"2024-01-15T10:30:01Z","level":"error","source":"api","message":"Payment failed","code":402}
    ]
  }'
POST /ingest/file Upload a log file via multipart/form-data

Supported formats: .json .jsonl .ndjson .csv .log .txt .gz. Format is auto-detected.

curl
curl -X POST https://yourapp.railway.app/ingest/file \
  -H "Authorization: Bearer oxk_your_key_here" \
  -F "file=@/var/log/myapp.log"

Rate Limits #

Rate limits are enforced per workspace per endpoint category. Exceeding a limit returns HTTP 429 Too Many Requests with a Retry-After header.

Endpoint groupLimitWindowNotes
POST /ingest/*300 req1 minBatch requests count as 1 regardless of event count
GET /api/logs120 req1 minReduce per_page to lower DB cost per request
POST /api/logs/:id/explain20 req1 minAI Explain calls Claude API — subject to Anthropic limits
Auth endpoints10 req15 minLogin, create-account, forgot-password
All other200 req1 minAlert, key, webhook CRUD endpoints
Rate limit response headers
HTTP Response Headers
X-RateLimit-Limit: 300
X-RateLimit-Remaining: 142
X-RateLimit-Reset: 1705316460    # Unix timestamp when window resets
Retry-After: 23                  # Only on 429 — seconds to wait

S3 Bucket Integration #

Connect ObserveX to any AWS S3 (or S3-compatible) bucket to automatically pull and index log files on a cron schedule.

Required IAM policy

Create an IAM user with programmatic access and attach this policy:

IAM Policy JSON
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:ListBucket"],
      "Resource": [
        "arn:aws:s3:::your-log-bucket",
        "arn:aws:s3:::your-log-bucket/*"
      ]
    }
  ]
}
Setup steps
01

Add the integration

Go to Integrations → Add integration → S3 Bucket. Enter bucket name, region, optional key prefix, and IAM credentials.

02

Create a scheduled job

Go to Ingestion Jobs → Create job, select your S3 integration, and set a cron schedule (e.g. every 15 minutes).

03

Monitor runs

Each run shows record count, status, duration, and error messages in Ingestion Jobs → Recent run history.

Verifying webhook delivery

ObserveX sends a X-ObserveX-Signature header on every webhook using HMAC-SHA256 of the request body. If you add a Webhook Secret on the alert rule, verify it server-side:

import hmac, hashlib

def verify_signature(secret: str, body: bytes, sig_header: str) -> bool:
    expected = hmac.new(secret.encode(), body, hashlib.sha256).hexdigest()
    return hmac.compare_digest(f"sha256={expected}", sig_header)
const crypto = require('crypto');

function verifySignature(secret, rawBody, sigHeader) {
  const expected = 'sha256=' + crypto
    .createHmac('sha256', secret)
    .update(rawBody)
    .digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(expected), Buffer.from(sigHeader)
  );
}

Scheduled Jobs #

Scheduled jobs automate ingestion from connected integrations (currently S3 Bucket). Jobs run on a cron schedule and are executed server-side on Railway.

Job lifecycle
StatusMeaning
pendingWaiting for the cron timer to fire
runningCurrently fetching and ingesting files
successCompleted; see records count in run history
errorFailed; error message available in run history detail
pausedManually paused — will not fire until resumed
ℹ
Jobs track the last processed file key and skip already-ingested objects. Duplicate ingestion is prevented via the S3 ETag + upload record deduplication check.

✦ AI Explain #

ObserveX integrates with Claude AI (Anthropic) to explain error and warning logs in plain English. Click any ERROR or WARN log in the explorer, then hit ✦ AI Explain in the detail panel.

What it explains
  • Root cause of the error in plain language
  • Likely contributing factors inferred from the payload
  • Suggested next steps to investigate or resolve
  • Related patterns to check in surrounding logs
ℹ
AI Explain requires an ANTHROPIC_API_KEY environment variable set on your Railway service. Add it in Railway → Your service → Variables.
Via API
POST /api/logs/:id/explain
curl -X POST https://yourapp.railway.app/api/logs/1042/explain \
  -H "Authorization: Bearer oxk_your_key_here"

# Response:
{
  "explanation": "This error indicates a TCP connection timeout to Stripe's API...",
  "log_id": 1042
}
