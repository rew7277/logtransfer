    🔑 API Reference

    ObserveX provides a REST API to ingest log events programmatically. All endpoints require a Bearer token from your API Keys page.

    Authentication

    Pass your API key in the Authorization header for every request.

    Authorization: Bearer oxk_your_api_key_here
    Endpoints
    MethodEndpointDescription
    POST/ingest/jsonIngest a single JSON log event
    POST/ingest/batchIngest up to 1000 events in one request
    POST/ingest/fileUpload a log file (multipart)
    GET/api/logsQuery logs with filter, search, pagination
    GET/api/logs/:idFetch single log event by ID
    DELETE/api/logs/:idDelete a single log event
    GET/api/bootstrapSummary stats, charts, alerts
    Ingest a single event
    POST /ingest/json Content-Type: application/json Authorization: Bearer oxk_your_key { "timestamp": "2024-01-15T10:30:00Z", "level": "error", "source": "payment-service", "message": "Stripe timeout after 30s", "requestId": "req_abc123", "userId": "usr_456" }
    Batch ingest
    POST /ingest/batch Content-Type: application/json Authorization: Bearer oxk_your_key { "events": [ {"timestamp": "...", "level": "error", "source": "...", "message": "..."}, {"timestamp": "...", "level": "info", "source": "...", "message": "..."} ] }
    Query parameters for GET /api/logs
    ParamTypeDescription
    pageintPage number (default: 1)
    per_pageintResults per page (max: 500)
    levelstringFilter: error, warn, info, success, debug
    qstringFull-text search in message field
    sourcestringFilter by source tag
    from_tsdatetimeStart of time range (ISO 8601)
    to_tsdatetimeEnd of time range (ISO 8601)
    sortstringColumn to sort by (timestamp, level, source)
    dirstringasc or desc (default: desc)
    🪣 S3 Bucket Guide

    Connect ObserveX to an AWS S3 bucket to automatically pull and index log files on a schedule.

    Prerequisites

    You need an AWS IAM user with the following permissions on your bucket:

    { "Effect": "Allow", "Action": [ "s3:GetObject", "s3:ListBucket" ], "Resource": [ "arn:aws:s3:::your-log-bucket", "arn:aws:s3:::your-log-bucket/*" ] }
    Supported log formats
    FormatExtensionDescription
    JSON Lines.jsonl, .ndjsonOne JSON object per line
    CloudWatch Export.jsonAWS CloudWatch log export format
    Plain text.log, .txtOne log line per row
    CSV.csvTabular log data with headers
    Compressed.gzGzip-compressed versions of above
    Setup steps

    1. Go to Integrations → Add integration and select S3 Bucket.

    2. Enter your bucket name, region, optional key prefix, and IAM credentials.

    3. Save the integration — ObserveX will validate access automatically.

    4. Go to Ingestion Jobs → Create job, select your new S3 integration, and set a cron schedule.

    Common cron schedules
    */15 * * * * → every 15 minutes 0 * * * * → every hour 0 0 * * * → daily at midnight UTC 0 */6 * * * → every 6 hours
    Key prefix filtering

    Use the prefix field to limit which objects are pulled. For example, prod/api-gateway/ will only ingest objects under that path.

    Troubleshooting
    ErrorCauseFix
    AccessDeniedMissing s3:GetObject permissionUpdate IAM policy
    NoSuchBucketWrong bucket name or regionVerify bucket & region
    InvalidCredentialsWrong Access Key/SecretRegenerate IAM key pair
    0 files foundPrefix doesn't match any objectsCheck prefix or leave blank
    🔔 Alert Configuration

    Alert rules continuously monitor your incoming logs and fire notifications when conditions are met. Alerts can be delivered via email or webhook (Slack, Teams, Discord, custom HTTP).

    Alert types
    TypeTrigger condition
    Error RateError count exceeds threshold in a rolling 5-minute window
    Keyword MatchA specific phrase appears in log messages ≥ threshold times
    Suspicious IPA single IP exceeds threshold req/min with high error rate
    Source Error SpikeA specific source emits more errors than threshold in 5 min
    Severity levels
    LevelMeaningSuggested use
    SEV3 – MediumElevated error rate or unusual patternSlack notification
    SEV2 – HighService degradation likelyEmail + Slack
    SEV1 – CriticalProduction outage or breach riskPagerDuty / phone
    Webhook payload format

    When a rule triggers, ObserveX sends a POST request to your webhook URL with this JSON body:

    { "alert_name": "Payment timeout spike", "severity": "sev2", "alert_type": "keyword", "condition": "keyword:\"Connect timeout\", threshold:5", "triggered_at": "2024-01-15T10:35:00Z", "match_count": 12, "org": "your-org", "link": "https://yourapp.railway.app/workspace/your-org/explorer" }
    Slack webhook example

    Create an Incoming Webhook in your Slack app settings, then paste the URL into the Webhook field when creating an alert. ObserveX will automatically format the message for Slack.

    Cooldown period

    The cooldown field (in minutes) prevents alert fatigue by suppressing repeated notifications for the same rule. Set to 0 for no cooldown (fires every time the condition is met). Recommended: 15–60 minutes for production alerts.

    Preview match count

    Before saving an alert rule, click Preview match count to see how many existing logs would trigger this rule. This helps you tune the threshold before going live.
